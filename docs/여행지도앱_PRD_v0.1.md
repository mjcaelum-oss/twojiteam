## 1. 개요

**한 줄 정의**
가고 싶은 지역이나 장소를 검색하면, 취향·여행 속도·동선을 반영해 관광지를 하나씩 추천하고, 선택한 곳들을 지도 위 실행 가능한 여행 일정으로 완성해주는 웹앱.

**왜 만드는가**
기존 여행 정보 서비스는 "추천"과 "계획"이 분리돼 있다. 블로그·SNS에서 장소를 모으고, 지도 앱에 하나씩 찍고, 운영시간을 각각 확인하고, 순서를 엑셀이나 메모장으로 정리한다. 이 앱은 **추천 → 선택 → 동선 → 검증 → 공유**를 한 흐름 안에서 끝낸다.

**차별점 (제품의 핵심 가설)**
1. **한 번에 하나씩 고르는 추천** — 리스트 20개를 던지지 않고, 3개 후보 중 하나를 고르게 해 결정 피로를 줄인다.
2. **동선을 아는 추천** — 이미 담은 장소로부터의 거리를 점수에 반영해, 물리적으로 말이 되는 조합만 제안한다.
3. **닫힌 문 앞에서 멈추지 않는 계획** — 도착 예상시각과 운영시간을 대조해 미리 경고한다.

## 3. 목표 및 성공 지표

### 3.1 제품 목표 (MVP)
- 검색 → 계획 완성까지 **5분 이내**에 도달 가능
- 완성된 계획이 **실행 가능**할 것 (운영시간·이동시간 검증 통과)

### 3.2 성공 지표

| 지표 | 정의 | 목표 |
|---|---|---|
| 계획 완성률 | 검색 시작 → "계획 완료" 도달 비율 | ≥ 40% |
| 평균 담은 장소 수 | 완성 계획당 장소 수 | ≥ 3곳 |
| 추천 수용률 | 제시된 후보 중 담기 선택 비율 | ≥ 30% |
| 공유·저장률 | 완성 계획 중 공유/저장 실행 비율 | ≥ 25% |
| 경고 발생률 | 운영시간·초과시간 경고가 뜬 계획 비율 | 모니터링용 |

## 4. 타깃 사용자

**주 타깃 — "느슨한 계획러"**
국내 1~2일 여행을 앞두고, 대략의 목적지(도시)는 정했으나 세부 코스는 미정. 블로그 검색은 귀찮고, 그렇다고 무계획으로 가긴 불안하다. 20~30대 커플·친구 여행이 중심.

**시나리오**
> 주말에 경주에 가기로 했다. "경주"를 검색한다. 문화·역사 / 알차게 / 커플을 고른다.
> 불국사 → 첨성대 → 황리단길 순으로 추천을 받아 담는다.
> "첨성대 도착 예상 17:40 — 이날 18:00 마감입니다" 경고를 보고 순서를 바꾼다.
> 완성된 일정을 카카오톡으로 동행에게 보낸다.

## 5. 범위 (Scope)

### 5.1 MVP 포함
- 장소·지역 **검색**(Google Places Autocomplete) 기반 시작
- 취향 3종(관광방식 / 속도 / 동행) 입력
- **거리·속도를 반영한** 순차 추천 (3개 후보 카드)
- 지도 위 후보·선택 장소 표시
- 구간별 이동수단 선택 및 경로 표시
- 날짜·출발시각 입력 → **도착 예상시각 계산 → 운영시간 검증**
- 리뷰 화면에서 **삭제 / 순서변경 / 구간 수정**
- 총 소요시간(체류+이동), 총비용(입장료+이동비) 합계
- localStorage 저장 + 공유 링크

### 5.2 MVP 제외 (Phase 2 이후)
- 회원가입·로그인, 서버 저장
- 숙소·맛집 예약 연동
- 다인원 협업 편집(동행이 함께 수정)
- Day 3 이상 다일정 최적화
- 리뷰·평점 작성
- 네이티브 앱

## 7. 기능 요구사항

우선순위: **P0** = MVP 필수 / **P1** = MVP 권장 / **P2** = 이후

### FR-01. 지역·장소 검색 `P0`
- Google Places Autocomplete로 지역(도시) 및 POI 검색
- **도시 입력** → 해당 도시 bounds를 추천 범위로 설정
- **POI 입력** → 해당 좌표 반경 N km(기본 10km, 조정 가능)를 추천 범위로 설정
- 검색 지점의 성격(여행 대상 지역 / 출발지)을 선택하게 함
- 검색 결과 없음 / 해외 지역 입력 시 안내 메시지

**Acceptance**
- "경주" 입력 시 경주 시내 범위의 관광지가 추천된다
- "불국사" 입력 시 불국사 반경 10km 관광지가 추천된다
- Autocomplete에 **세션 토큰**을 적용해 과금이 요청 단위로 집계된다

### FR-02. 여행 조건 입력 `P0`
- 관광방식: 문화·역사 / 자연·풍경 / 맛집·카페 / 체험·액티비티
- 여행 속도: 여유롭게 / 균형 있게 / 알차게 → **§8.2 파라미터로 연결**
- 동행 유형: 커플 / 혼자 / 가족 / 친구
- **여행 날짜**(date) 및 **출발 시각**(time) — 신규
- **인원수**(1~10) — 신규 (비용 합계에 사용)

### FR-03. 추천 엔진 `P0`
- §8 스펙에 따라 후보를 점수화하고 상위 3개를 제시
- 각 카드에 **추천 근거**를 노출: 매칭된 태그 + "직전 장소에서 12km"
- 담기 / 거절한 장소는 후보 풀에서 **제거**
- "셋 다 별로예요" → 다음 3개 후보 제시
- 후보 소진 시 "이 지역 추천을 모두 살펴봤어요" 상태 표시

**Acceptance**
- 한 번 노출 후 거절한 장소는 다시 나타나지 않는다
- 이미 담은 장소는 후보에 나타나지 않는다
- 직전 장소에서 왕복 이동시간이 속도 파라미터의 상한을 넘는 장소는 후보에서 제외되거나 강한 감점을 받는다

### FR-04. 지도 표시 `P0`
- 후보 3곳을 **서로 다른 색 핀**으로 동시 표시 (선택 전에도)
- 담은 장소는 순번이 찍힌 핀으로 강조
- 담은 장소 2곳 이상 시 구간 경로(Polyline) 표시
- 핀 클릭 시 InfoWindow (이름 · 지역 · 예상 체류시간)

### FR-05. 이동수단·경로 `P0`
- 구간별 이동수단: 자동차 / 대중교통 / 도보
- **이동수단 선택은 리뷰 화면에서 수행** (추천 화면에서는 "담기"에만 집중)
- 기본값: 구간 거리에 따라 자동 추천 (도보 < 1.5km, 그 외 자동차)
- Directions API로 구간별 소요시간·거리 산출
- 경로 계산 실패 시 해당 구간에 "경로 없음" 표기하되 계획 전체는 유지

### FR-06. 시간 검증 `P0` ★ 핵심
- 출발시각 + Σ(이동시간 + 체류시간)으로 **각 장소의 도착·출발 예상시각** 계산
- 각 장소의 운영시간·휴무일과 대조
  - 도착 예상시각이 마감 이후 → 🔴 "도착 시 이미 마감입니다"
  - 마감 1시간 이내 도착 → 🟡 "체류시간이 부족할 수 있어요"
  - 해당 요일 휴무 → 🔴 "이 날은 휴무일입니다"
- 경고 시 **해결 액션 제시**: 순서 바꾸기 / 이 장소 빼기 / 출발 시각 앞당기기

**Acceptance**
- 화요일에 경복궁을 담으면 휴무 경고가 뜬다
- 마감 이후 도착이 예상되면 해당 카드가 시각적으로 구분된다

### FR-07. 계획 편집 `P0`
- 리뷰 화면에서 장소 **삭제**
- 장소 **순서 변경** (드래그 앤 드롭 또는 ↑↓ 버튼)
- 구간 이동수단 변경
- 편집 시 경로·시간·비용·경고가 즉시 재계산

### FR-08. 요약 정보 `P0`
- **총 소요시간** = Σ 체류시간 + Σ 이동시간 (기존엔 체류만 합산)
- **총 비용** = Σ 입장료(× 인원수) + Σ 이동비
- 장소 수, Day 수
- 각 항목이 "추정치"임을 명시

### FR-09. Day 자동 분할 `P1`
- 하루 가용시간(기본 09:00~20:00, 설정 가능)을 초과하면 Day 2로 분할 제안
- Day 경계에서 "숙소 위치를 정하시겠어요?" 안내 (Phase 2 연결점)

### FR-10. 저장·공유 `P1`
- localStorage 자동 저장 (새로고침 시 복원)
- 계획을 URL 인코딩한 **공유 링크** 생성 → 카카오톡·메시지로 전송 가능
- 이미지 또는 PDF로 내보내기 `P2`

### FR-11. 장소 사진 `P1`
- 각 후보 카드와 리뷰 항목에 대표 사진 1장 표시
- Places photo reference 또는 TourAPI `firstimage` 사용
- 사진 없을 시 카테고리별 플레이스홀더

## 8. 추천 알고리즘 스펙

### 8.1 점수 공식

```
score = W_style × styleMatch      // 취향 일치
      + W_comp  × companionMatch  // 동행 일치
      + W_pop   × popularity      // 인기도(리뷰 수·평점 정규화)
      - W_dist  × distancePenalty // 동선 이탈 감점  ← 신규
```

| 항목 | 계산 | 가중치(초안) |
|---|---|---|
| styleMatch | 관광방식 태그 일치 시 1 | 5 |
| companionMatch | 동행 태그 일치 시 1 | 2 |
| popularity | 0~1 정규화 | 2 |
| distancePenalty | `min(1, travelTime / maxLegTime)` | **6** |

- **기준점(anchor)**: 담은 장소가 없으면 검색 좌표, 있으면 **직전에 담은 장소**
- `travelTime`은 직선거리 기반 근사치 사용(Directions 호출 최소화), 담긴 후 정확 계산으로 대체
- `travelTime > maxLegTime`인 후보는 **후보 풀에서 제외**

### 8.2 여행 속도의 실질 반영 ★ 신규

속도는 **거리 상한**과 **하루 소화량**을 결정한다.

| 속도 | maxLegTime (구간 최대 이동시간) | 하루 목표 장소 수 | 체류시간 배수 | 추천 성향 |
|---|---|---|---|---|
| **여유롭게** (slow) | 30분 | 2~3곳 | ×1.3 | 가까운 곳 위주, 체류 긴 장소 선호 |
| **균형 있게** (balanced) | 50분 | 3~4곳 | ×1.0 | 기본값 |
| **알차게** (fast) | 90분 | 5~6곳 | ×0.8 | 먼 명소도 허용, 짧게 여러 곳 |

- 하루 목표 장소 수에 도달하면 "오늘 일정이 꽉 찼어요. 계획을 마칠까요?" 제안
- 체류시간 배수는 도착시각 계산과 총 소요시간에 반영

## 9. 데이터 요구사항

### 9.1 데이터 소스 ⚠️ **미결정 — 최우선 결정 사항**

| 안 | 장점 | 단점 |
|---|---|---|
| **A. TourAPI** | 국내 관광지 큐레이션 품질, 무료, 운영시간·입장료 제공 | 커버리지 편차, 데이터 최신성 |
| **B. Google Places** | 전국 커버리지, 사진·평점·영업시간 | 과금, "관광지" 필터링 어려움 |
| **C. 하이브리드 (권장)** | Places로 후보 수집 → TourAPI 상세·자체 태그로 보강 | 구현 복잡도, 매칭 로직 필요 |

> **이 결정 전까지 FR-01/FR-03의 구현을 시작할 수 없다.**

### 9.2 장소 스키마

```ts
interface Spot {
  id: string;
  name: string;
  region: string;          // "경북 경주시"
  lat: number; lng: number;
  category: 'culture' | 'nature' | 'food' | 'activity';
  tags: string[];          // companion 등 보조 태그
  description: string;
  photoUrl?: string;       // FR-11
  fee: { amount: number; note: string };   // 인원수 곱셈 위해 숫자 필요
  duration: number;        // 표준 체류시간(분)
  hours: OpeningHours;     // 요일별 open/close + 휴무일  ← 구조화 필수
  popularity: number;      // 0~1
  source: 'tourapi' | 'places';
  sourceUrl: string;
  lastVerifiedAt: string;  // 데이터 확인일 표기 의무
}

interface OpeningHours {
  weekly: { [day: 0..6]: { open: 'HH:mm'; close: 'HH:mm' } | null }; // null=휴무
  closedDates?: string[];  // 특정 휴관일
  note?: string;
}
```

> 기존 프로토타입의 `hours: '09:00~18:00(계절별 변동)'` 같은 **문자열은 검증에 쓸 수 없다.** 구조화가 FR-06의 전제조건.

### 9.3 비용 데이터
- 입장료: 숫자 + 단위(성인 기준) 필수
- 자동차 이동비: 현재 `180원/km` 휴리스틱 → **"추정치(연료비만, 통행료·주차 미포함)"** 라벨 필수
- 대중교통: Directions `fare` 사용, 없으면 "요금 확인 필요"
